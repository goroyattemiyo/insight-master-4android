# InsightMasterSnap - 開発ルール

最終更新: 2026-03-31

---

## 1. 絶対ルール

1. **コード即書き禁止**: コードを書く前に必ず有識者議論（セクション2）を実施する
2. **要件優先**: 実装判断は常に `docs/REQUIREMENTS.md` と `docs/MVP_DEFINITION.md` を優先する
3. **MVP逸脱禁止**: MVP対象外の機能を勝手に実装しない
4. **コード保全**: 既存の動作するコードを許可なく大規模リファクタリング・削除・名前変更しない
5. **バイアス禁止**: 技術選定・設計判断に個人的好みを持ち込まない
6. **品質妥協禁止**: 動くが汚いコードは許可しない
7. **技術的異議の明示**: 問題がある場合は根拠付きで反論する
8. **エビデンスベース**: テスト結果・ビルド結果・UI確認など客観的指標で判断する
9. **スコープ厳守**: フェーズ外の要求は `docs/BACKLOG.md` に追記し、現タスクに集中する
10. **ハルシネーション禁止**: ファイルパス・関数名・設定値は実在確認してから使う
11. **ブランド統一**: 画面内・文書内のブランド名は必ず `InsightMasterSnap` に統一する
12. **旧名称禁止**: `The Insight Editorial` は使用しない
13. **API不要主軸**: 初期プロダクトは API連携型ではなく、スクリーンショット解析型を主軸とする
14. **初期対応SNS固定**: 初期対応は `Threads / Instagram / X` とする
15. **信頼性優先**: 完全自動をうたわず、数値修正可能な設計を維持する

## 2. 有識者議論プロセス

### 2.1 基本原則
すべての実装作業の前に、有識者を召喚し多角的に議論してから進めること。  
「とりあえず書く」「動けばよい」は禁止。

### 2.2 有識者の召喚
作業内容に応じて、最低3名以上の専門家ペルソナを召喚する。

| ペルソナ | 視点 | 召喚する場面 |
|----------|------|-------------|
| アーキテクト | 全体設計・依存関係・拡張性 | 新機能追加・構成変更 |
| Androidエンジニア | Capacitor連携・端末制約・配布 | Android実装 |
| Webフロントエンド専門家 | UI構造・状態管理・保守性 | 画面追加・UI改修 |
| OCR/画像解析担当 | 画像読取精度・補正設計 | 数値抽出・画像解析 |
| UXデザイナー | 初心者体験・片手操作・分かりやすさ | UI変更・導線設計 |
| QAエンジニア | テスト容易性・回帰リスク・エッジケース | テスト設計・修正 |
| セキュリティ専門家 | 権限・画像データ・保存方針 | 保存・共有・権限 |
| ビジネスアナリスト | MVP優先度・価値訴求・競合観点 | 機能要否の判断 |

### 2.3 議論の進め方
1. 課題定義
2. 有識者召喚
3. 多角的議論
4. スコア項目策定
5. 100点満点でスコアリング
6. 80点以上で着手許可
7. 議論結果を `docs/DECISIONS.md` に記録してから実装開始

### 2.4 スコアリング基準
基本項目を以下とし、必要に応じて観点を追加する。

| # | 評価項目 | 配点目安 | 観点 |
|---|----------|----------|------|
| 1 | 技術的妥当性 | /20 | 技術選定は適切か |
| 2 | 保守性・可読性 | /15 | 将来の修正がしやすいか |
| 3 | スコープ適合性 | /15 | MVPや現フェーズに合っているか |
| 4 | UX適合性 | /15 | 初心者に分かりやすいか |
| 5 | テスト容易性 | /10 | テストや検証がしやすいか |
| 6 | リスク・副作用 | /10 | 既存機能への悪影響は小さいか |
| 7 | 費用対効果 | /10 | 実装コストに見合う価値があるか |
| 8 | Android適合性 | /5 | Android上で無理なく成立するか |
| | **合計** | **/100** | **80点以上で着手許可** |

### 2.5 80点未満の場合
- 低スコア項目の改善案を再検討する
- スコープを縮小する
- 代替案を比較する
- それでも80点未満なら見送り、理由を `docs/DECISIONS.md` に記録する

## 3. セッション開始プロトコル

### 3.1 新規会話の開始時
毎回、以下を優先して確認する。

**必須**
1. `docs/RULES.md`
2. `docs/REQUIREMENTS.md`
3. `docs/MVP_DEFINITION.md`
4. `docs/DEVELOPMENT_PLAN.md`

**必要時のみ**
5. `docs/DECISIONS.md`
6. `docs/ARCHITECTURE.md`
7. `docs/UI_UX_GUIDELINES.md`
8. `docs/DIRECTORY_STRUCTURE.md`

### 3.2 会話圧縮時
同じ手順で再確認する。記憶ではなく、docsを正とする。

### 3.3 セッション終了時
必要に応じて以下を更新する。
- `docs/DEVELOPMENT_PLAN.md`
- `docs/DECISIONS.md`
- `docs/BACKLOG.md`
- `docs/RULES.md` の監視テーブル

## 4. トークン節約ルール

### 4.1 基本原則
会話が長くなるほどコストは増える。  
毎ターン送る情報を最小化する。

### 4.2 ドキュメントの3層構造

| 層 | 内容 | 読み込み | サイズ上限 |
|----|------|----------|-----------|
| Tier 1 | RULES, REQUIREMENTS, MVP_DEFINITION, DEVELOPMENT_PLAN | 毎セッション優先 | 各15KB以下 |
| Tier 2 | DECISIONS, ARCHITECTURE, UI_UX_GUIDELINES, DIRECTORY_STRUCTURE | 必要時のみ | 各15KB以下 |
| Tier 3 | CHANGELOG, archive, 詳細メモ | リンク参照のみ | 常時読まない |

### 4.3 入力削減
- ログは必要箇所のみ抜粋
- スクショは必要時のみ使う
- ファイル全文貼りは避け、該当部分を優先する
- 大型生成物や build 生成物は読ませない

### 4.4 出力削減
- 同種の質問はまとめる
- 実装時は必要な差分に集中する
- 長い説明より決定事項を優先する

---

## 5. 変更前の影響範囲宣言

変更前に以下を宣言すること。

1. 変更対象ファイル
2. 間接影響ファイル
3. テストへの影響
4. ロールバック方法

5ファイル以上へ影響する場合は、事前に人間の承認を得る。

---

## 6. ハルシネーションガード

- ファイルパスは実在確認する
- 関数名・クラス名は検索確認する
- 設定値は設定ファイルやdocsで確認する
- 依存パッケージは manifest / package / build 定義で確認する

未確認の想定でコードを書かない。

---

## 7. スコープロック & バックログ規律

### 7.1 現タスク以外の変更禁止
ついで修正は禁止。見つけた改善案は `docs/BACKLOG.md` に記録する。

### 7.2 BACKLOG形式

| 日付 | 提案内容 | 発生経緯 | 優先度 | 状態 |
|------|----------|----------|--------|------|
| YYYY-MM-DD | 内容 | どのタスク中に発見 | 高/中/低 | 未着手 |

---

## 8. 決定ログ（DECISIONS.md）

### 8.1 目的
会話が消えても、なぜその判断をしたかを残す。

### 8.2 記録形式

**## D-001: [タイトル]**
- 日付: YYYY-MM-DD
- 背景:
- 選択肢:
- 決定:
- 理由:
- スコア:

---

## 9. ファイルサイズ制限

| 対象 | 上限 |
|------|------|
| ソースコード（.ts, .tsx, .js, .kt, .swift 等） | 30 KB |
| ドキュメント（.md） | 15 KB |
| 設定ファイル（.json, .yaml, .xml 等） | 10 KB |

- 上限超過時は分割する
- Tier 1 ドキュメントは特に短く保つ
- 例外時は理由をコミットメッセージに残す

---

## 10. ワークフロールール

- push後は必ずリポジトリから直接確認する
- main または明確な作業ブランチで作業する
- コミットメッセージは `feat: fix: docs: test: refactor:` を使う
- コミット前に可能な範囲で lint / test / build を通す

---

## 11. テストルール

- 新規ロジックには可能な限りテストを追加する
- 最低限、以下を確認する
  - Web build
  - 単体テスト
  - Capacitor sync
  - Android debug build
- テスト観点は「純粋ロジック / UI / Android連携 / 手動E2E」で分ける
